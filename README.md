# qubots-leaderboard

> A community-curated leaderboard for [qubots](https://github.com/leonidas1312/qubots) — pluggable optimization problems and solvers.

Every solver in `submissions/` runs against every benchmark in `benchmarks/`. Results are auto-generated by GitHub Actions on every merge and on a weekly cron, and committed back to this repo as [LEADERBOARD.md](LEADERBOARD.md) and [leaderboard.json](leaderboard.json).

The leaderboard is intentionally git-native: every submission is a YAML in `submissions/`, every result is a commit, and anyone can reproduce the run locally with `qubots leaderboard`. There's no hosted service, no login, no central authority — just a maintained repo and a CI runner.

## Submit your solver (10 minutes)

1. **Make sure your solver is a valid qubots component**: a public GitHub repo with a `qubots.yaml` (declaring `type: optimizer` and any `requirements:`) and a `qubot.py` entrypoint. New to this? Run `qubots new optimizer --name my_solver --flavor blackbox` to scaffold one.
2. **Pin a commit**: get the full SHA you want benchmarked (`git rev-parse HEAD`).
3. **Open a PR adding one file** under `submissions/<your-username>/<solver-name>.yaml`:

   ```yaml
   qubots_submission_schema_version: 1
   spec: github:your-username/your-repo@<full-sha>:path/to/component
   submitter: your-username
   display_name: "My Solver (default config)"
   parameters:
     time_limit_seconds: 30
   ```

4. **CI runs** automatically on PR open. The workflow installs your solver's declared `requirements:` into an isolated venv, runs `qubots leaderboard` against the canonical suite, and posts the results as a comment on your PR.
5. **A maintainer reviews** (mostly: is the spec parseable, is the solver reasonable, are the dependencies sane). On merge, CI runs again and commits the updated leaderboard back to `main`.

See [CONTRIBUTING.md](CONTRIBUTING.md) for the full guide, including how to add a new benchmark.

## Run the leaderboard locally

```bash
pip install qubots[highs,cpsat]
qubots leaderboard \
  --suite benchmarks/flagship.yaml \
  --submissions submissions/ \
  --out LEADERBOARD.md \
  --json leaderboard.json
```

## Why a leaderboard for OR

The OR community already lives on [MIPLIB](https://miplib.zib.de) and the [MiniZinc Challenge](https://www.minizinc.org/challenge) for benchmarking — what's missing is a modern, git-native, community-driven layer that's easy to contribute to. qubots-leaderboard wires existing benchmark instances + community-published solver components into one reproducible pipeline. Contributing a solver doesn't require a paper, an institution, or an account — just a public repo and a PR.

## Curated, not crowd-sourced

A maintainer reviews every submission before merge. The bar is:
- Spec resolves and the component validates (`qubots validate <spec> --trust-remote-code`).
- Declared `requirements:` are reasonable (no commercial-license-required deps without the user opting in; no kitchen-sink dependency lists).
- Solver doesn't try to bypass the harness or cheat on benchmark instances.

Submissions that go stale (upstream repo deleted, dependencies broken) are marked `archived` rather than deleted, so historical leaderboards remain reproducible.
